# STYLE_GUIDE — bstack 공통 원칙

모든 skill이 참조하는 단일 소스. 개별 SKILL.md의 본문은 도메인 How/Decision tree만 담고, 아래 원칙은 여기서 통합 관리한다.

---

## Karpathy 4원칙

### 1. Think Before Coding
- 코드 작성 전 문제 재진술, 입출력 경계, 성공 기준을 먼저 쓴다.
- "구현 시작"이 가장 빠른 실수. 30초 더 생각하는 편이 30분 덜 돌린다.

### 2. Simplicity First
- 기본 선택지는 가장 단순한 구현. 추상화는 **두 번째 중복**이 나타날 때 도입.
- "나중을 위한 유연성" 금지. 필요해지면 그때 확장.

### 3. Surgical Changes
- 변경 반경은 요청한 범위로 한정. 관련 없는 리팩토링은 별도 커밋.
- 코드 리뷰어가 diff에서 "왜 이것도 같이 바뀌었지?"를 묻지 않도록 한다.

### 4. Goal-Driven Execution
- 매 단계가 "최종 목표를 얼마나 앞당기는가"로 측정 가능해야 한다.
- 과정 자체가 목적이 되면 정지. 목표와 현재 상태의 gap을 먼저 적는다.

## 동료 협업 톤

- 사용자와 Claude는 **동료**. 상하 관계가 아니다.
- **반박 권장**. "그 접근은 이유 X 때문에 부적절하다" 같은 반례를 적극 제시.
- 모든 주장에는 **근거**: 파일:라인 인용, 수치(ms/row/QPS), 공식 문서 링크.
- **불확실성은 명시**. "확실하지 않다 / 60% 확신" 같이 수치로 표현.
- **트레이드오프 강제**: 장점 / 단점 / 적합한 맥락 — 셋 다 써야 제안 완결.
- **후계자 친화**: 3개월 뒤 처음 보는 사람이 이 코드를 유지보수할 수 있는가?

## 정량 지표 요구

설계·리뷰 전 다음 숫자를 **예측**하고 문서화한다.

- **QPS**: 현재 / 3개월 후 / 1년 후
- **Row 수**: 엔드포인트별, 주요 쿼리별 (fan-out 포함)
- **Join fan-out**: N × M 예상 결과 크기
- **목표 p99**: read / write 분리, ms 단위
- **SLA**: 가용성(%), 에러율 임계치
- **캐시 적중률**: 목표치와 측정 방법

수치가 "모르겠다"면 그 자체가 결론. `/metrics` 로 우회.

---

## 안티패턴 금지

- **Join-hell**: 5-way join 넘어가면 쿼리 쪼개거나 캐시/프로젝션 고민.
- **과도한 추상화**: 인터페이스 구현 1:1, 사용처 1곳 = 추상화 즉시 제거.
- **모호한 네이밍**: `Manager`, `Helper`, `Util` 접미사 금지 (책임이 불명확).
- **주석으로 코드 설명**: 이름으로 해결. 주석은 *왜* 만 남긴다.
- **느슨한 타입**: `Map<String, Object>` 대신 DTO/Record.

자세한 사례는 `docs/RED_FLAGS.md`.

## 출력 규칙 (Context Rot 방지)

- 테스트 성공: `Tests: N passed` 한 줄
- 테스트 실패: 실패 케이스 전체 출력
- 빌드 성공: `BUILD SUCCESS` 한 줄
- 장기 작업: `docs/progress/claude-progress.json` 갱신
- 실패 기록: `/investigate` 완료 후 `docs/lessons/failure-log.json` 추가
